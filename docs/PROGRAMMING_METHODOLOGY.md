## 核心概念对比

### 定义

| 方法论 | Vibe Coding | Spec Coding |
|--------|------------|-------------|
| **核心哲学** | 直觉驱动 | 规范驱动 |
| **开发流程** | 即兴、探索性 | 结构化、有序化 |
| **文档重心** | 代码即文档 | 先文档后代码 |
| **变更管理** | 频繁、自由 | 受控、版本化 |
| **团队协作** | 个人或小团队 | 团队协作标准 |

### 详细对比

```
┌─────────────────────────────────────────────────────────────┐
│                    Vibe Coding                               │
├─────────────────────────────────────────────────────────────┤
│  💭 凭直觉和感觉编写代码                                      │
│  🎨 重视代码的美感和艺术性                                    │
│  🔧 随时调整、快速迭代                                          │
│  📝 最小化文档，代码自解释                                        │
│  🎯 适合：原型、创意项目、个人项目                                │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                    Spec Coding                                │
├─────────────────────────────────────────────────────────────┤
│  📋 遵循规范和需求编写代码                                      │
│  📊 先定义再开发                                                  │
│  🔍 强调可验证和可测试                                          │
│  📚 文档优先，版本控制                                            │
│  🎯 适合：企业项目、团队协作、长期维护                            │
└─────────────────────────────────────────────────────────────┘
```

### 决策机制对比

| 决策维度 | Vibe Coding | Spec Coding |
|---------|------------|-------------|

## 开发全流程对比

### 需求理解阶段

| 阶段 | Vibe Coding | Spec Coding |
|------|------------|-------------|
| **需求获取** | 口头沟通、即时理解 | 文档化需求、需求评审 |
| **需求分析** | 脑内构思、快速理解 | 结构化分析、需求建模 |
| **需求确认** | 边做边确认 | 需求签字、基线锁定 |
| **变更处理** | 随时接受 | 变更流程、影响分析 |
| **典型耗时** | 几分钟到几小时 | 几天到几周 |

### 设计阶段

| 设计类型 | Vibe Coding | Spec Coding |
|---------|------------|-------------|
| **架构设计** | 脑内构想或简单草图 | 详细架构文档、UML 图 |
| **接口设计** | 边写边定义 | API 规范、接口文档 |
| **数据设计** | 即时创建数据结构 | 数据模型、ER 图、Schema |
| **错误处理** | 遇到再加 | 预先定义错误码、异常处理 |
| **典型耗时** | 几乎跳过 | 几天 |

#### Vibe Coding 编码过程

```
时间线示例（开发一个用户认证功能）：

Day 1 Morning:
├── 09:00 - 开始写代码，直接创建 AuthController
│           "应该用 JWT，直接开干"
├── 10:30 - 写完登录接口
│           "看起来不错"
├── 11:00 - 测试时发现需要刷新 token
│           "加个 refresh endpoint"
└── 12:00 - 中午，功能基本能用

Day 1 Afternoon:
├── 14:00 - 觉得代码结构有点乱
│           "重构一下，把验证逻辑抽出来"
├── 16:00 - 发现安全问题
│           "需要加密码盐值"
└── 18:00 - 功能完成，但代码改了好多遍

特点：
✓ 快速产生可用代码
✓ 即时反馈和调整
✗ 代码需要多次重构
✗ 可能遗漏重要细节
```

#### Spec Coding 编码过程

```
时间线示例（开发同样的用户认证功能）：

前期准备（已完成）：
├── 需求文档：详细的用户故事、验收标准
├── 设计文档：架构图、API 规范、安全要求
├── 任务清单：分解为 15 个子任务
└── 估时：预计 2 天

Day 1 Morning:
├── 09:00 - 按任务清单开始
│           Task-1: 创建数据模型（符合规范）
├── 10:00 - Task-2: 实现密码加密
│           使用规范中指定的 bcrypt + salt
├── 11:00 - Task-3: 实现 JWT 生成
│           严格按照 API 规范
└── 12:00 - 完成 3 个任务，进度符合预期

Day 1 Afternoon:
├── 14:00 - Task-4~6: 实现登录/注册/刷新接口
│           完全符合 API 文档定义
├── 16:00 - Task-7~9: 单元测试
│           验收标准逐项验证
└── 18:00 - Day 1 完成，所有测试通过

Day 2:
├── 09:00 - Task-10~12: 集成测试、文档
└── 14:00 - Task-13~15: 代码审查、部署

特点：
✓ 按计划执行，进度可预测
✓ 代码质量稳定，重构少
✗ 需要前期投入大量时间
✗ 需求变更时调整成本高
```

### 测试阶段

| 测试类型 | Vibe Coding | Spec Coding |
|---------|------------|-------------|

| **集成测试** | 手动测试为主 | 自动化测试套件 |

| **性能测试** | 感觉慢了才测 | 按性能指标验证 |

### 维护阶段

| 维护场景 | Vibe Coding | Spec Coding |
|---------|------------|-------------|
| **Bug 修复** | 靠记忆和注释查找 | 通过规范和测试定位 |
| **功能扩展** | 看代码理解意图 | 查阅设计文档 |
| **人员交接** | 口头传授 + 代码注释 | 文档 + 规范 + 代码 |

## Spec Coding 内部分解

### Spec Encoding vs Open Coding

在规范驱动开发中，有两个重要的编码概念：

#### Spec Encoding（规范编码）

```
定义：将业务需求转换为技术规范的过程

输入：
├── 业务需求（自然语言）
├── 用户故事
└── 验收标准

处理：
┌──────────────────────────────────────────────────────────┐
│              Spec Encoding Process                       │
├──────────────────────────────────────────────────────────┤
│  1. 需求理解：                                            │
│     "用户想要上传 PDF 并与文档对话"                       │
│     → 解析：哪些用户？什么类型 PDF？对话方式？            │
│                                                              │
│  2. 领域建模：                                            │
│     • Actor: 用户、Admin、系统                            │
│     • Entity: PDF、文档、对话、消息                        │
│     • Action: 上传、解析、查询、回答                       │
│     • Data: 文件流、文本、向量、元数据                    │
│                                                              │
│  3. 技术映射：                                            │
│     • PDF 上传 → POST /api/upload                         │
│     • 文件验证 → filetype, size 检查                       │
│     • 文本解析 → pdf-parse 库                            │
│     • AI 对话 → LangChain + RAG                          │
│                                                              │
│  4. 规范定义：                                            │
│     • API 签名：request/response 结构                     │
│     • 约束条件：10MB 限制、超时时间                       │
│     • 验收标准：可度量、可验证                            │
└──────────────────────────────────────────────────────────┘

输出：
├── 技术规范文档 (spec.md)
├── API 接口定义
├── 数据模型设计
└── 验收标准清单

关键能力要求：
• 需求理解能力：听懂业务语言
• 抽象能力：从具体到抽象
• 技术视野：了解可用技术方案
• 沟通能力：与业务/技术双方对话
```

#### Open Coding（开放编码）

```
定义：将技术规范转换为可执行代码的过程

输入：
├── 技术规范 (spec.md)
├── 架构设计 (plan.md)
└── 任务列表 (tasks.md)

处理：
┌──────────────────────────────────────────────────────────┐
│              Open Coding Process                          │
├──────────────────────────────────────────────────────────┤
│  1. 规范解析：                                            │
│     "需要 PDF 上传 API，支持文件验证"                     │
│     → 提取：POST /api/pdf/upload                          │
│     → 提取：验证 PDF 格式、≤10MB                          │
│     → 提取：返回 pdfId 或错误                              │
│                                                              │
│  2. 技术选型：                                            │
│     • 框架：Next.js App Router                           │
│     • 文件处理：formData API                              │
│     • 验证库：file-type + 自定义逻辑                      │
│     • 存储：临时文件系统 → Redis                          │
│                                                              │
│  3. 代码实现：                                            │
│     interface UploadRequest {                             │
│       file: File;                                         │
│     }                                                      │
│     export async function POST(req: Request) {           │
│       const formData = await req.formData();              │
│       const file = formData.get('file');                  │
│       // validation...                                    │
│       // storage...                                       │
│       return Response.json({ pdfId });                    │
│     }                                                      │
│                                                              │
│  4. 测试验证：                                            │
│     • 单元测试：验证函数逻辑                              │
│     • 集成测试：端到端流程                                │
│     • 边界测试：10MB 文件、非 PDF 文件                    │
└──────────────────────────────────────────────────────────┘

输出：
├── 可执行代码
├── 单元测试
├── 集成测试
└── API 文档（代码生成）

关键能力要求：
• 代码实现能力：语法、框架、工具
• 调试能力：定位和修复问题
• 测试能力：编写测试用例
• 代码质量：可读性、性能、安全
```

### 两者关系

```
┌─────────────────────────────────────────────────────────────┐
│                    Spec Coding 流程                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  业务需求                                                   │
│     │                                                       │
│     ├──────────────────────────────────────┐              │
│     ▼                                        Spec Encoding │
│  技术规范                                        │              │
│     │  (What & Why)                          │              │
│     ├──────────────────────────────────────┐              │
│     ▼                                          Open Coding  │
│  可执行代码                                        │              │
│     │  (How)                                 │              │
│     ▼                                        │              │
│  部署系统                                        │              │
│                                                             │
│  反馈循环：                                                  │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐   │
│  │ 验收测试    │───▶│ 发现问题    │───▶│ 规范调整    │   │
│  └─────────────┘    └─────────────┘    └──────┬──────┘   │
│                                              │             │
│                         ┌────────────────────┘             │
│                         ▼                                  │
│              重新 Spec Encoding / Open Coding               │
│                                                             │
│  关键点：                                                    │
│  • Spec Encoding 决定"做什么"和"为什么"                     │
│  • Open Coding 决定"怎么做"                                 │
│  • 两者相互独立，可以由不同角色完成                          │
│  • 反馈循环持续改进                                         │
└─────────────────────────────────────────────────────────────┘
```

## 风险与成本分析

### 时间成本对比

#### 项目生命周期总时间

```
假设开发一个中等复杂度功能（如用户认证系统）

┌─────────────────────────────────────────────────────────────┐
│                    Vibe Coding 时间线                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ▓▓▓░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░│
│  └─ 2 周 ──┘                                              │
```

```typescript
import { PDFUploadRequest, PDFUploadResponse } from '@/types/api'
import { validatePDFFile } from '@/lib/validation'
import { saveTempPDF } from '@/lib/storage'

export async function POST(req: Request): Promise<Response> {
  // Parse request according to API specification
  const formData = await req.formData()
  const file = formData.get('file') as File

  // Validate against specification requirements
  const validation = validatePDFFile(file, {
    maxSize: 10 * 1024 * 1024, // 10MB from spec
    allowedTypes: ['application/pdf']
  })

  if (!validation.isValid) {
    return Response.json({
      error: validation.error.code,
      message: validation.error.message
    }, { status: 400 })
  }

  // Generate unique identifier as per spec
  const pdfId = crypto.randomUUID()

  // Store temporary file according to storage design
  await saveTempPDF(pdfId, file)

  // Return response matching API contract
  const response: PDFUploadResponse = {
    success: true,
    data: { pdfId }
  }

  return Response.json(response)
}
```

```markdown
### 验收标准
- 支持的文件格式：application/pdf
- 文件大小限制：10MB
- 必须返回唯一的文件 ID
- 上传失败时返回友好的错误信息

### API 规范
POST /api/pdf/upload

Request:
- file: File (multipart/form-data)

Response 200:
{
  "success": true,
  "data": {
    "pdfId": "uuid"
  }

Response 400:
{
  "error": "INVALID_FILE_TYPE",
  "message": "只支持 PDF 格式"
```

**Step 2: Open Coding - 编写代码**

```typescript
// 基于规范实现代码
import { validatePDFFile } from '@/lib/utils/validation';
import { generateTempFileName } from '@/lib/storage/temp-files';

export async function POST(req: Request) {
  const formData = await req.formData();
  const file = formData.get("file") as File;

  // 验证（基于规范）
  const validation = validatePDFFile(file);
  if (!validation.valid) {
    return NextResponse.json(
      { error: validation.error },
      { status: 400 }
    );
  }

  // 生成唯一 ID
  const pdfId = randomUUID();
  const tempPath = generateTempFileName(pdfId);

  // 保存文件
  const buffer = await file.arrayBuffer();
  await saveTempFile(buffer, tempPath);

  return NextResponse.json({
    success: true,
    data: { pdfId, tempPath }
  });
}
```

**开发过程**：
- 30 分钟编写规范
- 1 小时编写代码

## 总结

### 方法对比矩阵

| 维度 | Vibe Coding | Spec Coding |
|------|------------|-------------|
| **核心哲学** | 直觉驱动 | 规范驱动 |
| **起点** | 代码即起点 | 规范即起点 |
| **编码过程** | Spec Encoding + Open Coding 混合 | 分离的 Spec Encoding 和 Open Coding |
| **文档** | 最小化 | 完整文档 |
| **适用项目** | 小型、创意、个人项目 | 大型、团队、长期项目 |
| **可维护性** | 低 | 高 |
| **学习曲线** | 低 | 中 |
| **工具支持** | 无 | Spec Kit 等工具 |
| **前期投入** | 极低 | 中等 |
| **长期成本** | 高 | 低 |
| **团队规模** | 1 人 | 2+ 人 |

### Spec Coding 中的两步编码

| 阶段 | 名称 | 输入 | 输出 | 能力要求 |
|------|------|------|------|---------|
| 第一阶段 | **Spec Encoding** | 业务需求 | 技术规范 | 需求分析、抽象思维 |
| 第二阶段 | **Open Coding** | 技术规范 | 可执行代码 | 代码实现、测试 |
