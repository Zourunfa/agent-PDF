Spec Coding 内部分解

Spec Encoding vs Open Coding

📝
Spec Encoding
规范编码：需求 → 规范

输入：业务需求（自然语言）

处理过程：

  • 需求理解：解析业务语言
  • 领域建模：提取实体和关系
  • 技术映射：选择技术方案
  • 规范定义：编写可执行规范

输出：技术规范文档

⚙️
Open Coding
操作编码：规范 → 代码

输入：技术规范、架构设计

处理过程：

  • 规范解析：提取技术要求
  • 技术选型：确定实现工具
  • 代码实现：编写可执行代码
  • 测试验证：单元和集成测试

输出：可执行代码 + 测试

最终建议

  • 个人项目、原型：使用 Vibe Coding，快速迭代
  • 团队项目、长期项目：使用 Spec Coding，投资前期，节省后期
  • 不确定时：选择 Spec Coding（更保险）
  • 从 Vibe 到 Spec：渐进式引入，不要一刀切
  • 工具助力：使用 Spec Kit 等工具降低 Spec Coding 的门槛

Spec-Kit 编程详细步骤

什么是 Spec-Kit？

Spec-Kit 是一套规范驱动开发（Spec-Driven Development）的工具和方法论，通过结构化的工作流程， 将模糊的业务需求转化为可执行的技术规范，最终生成高质量的代码。它让 Spec Coding 变得简单、可追溯、可协作。

📜 步骤 1: 创建项目宪法 (Constitution)

/speckit.constitution 一次性初始化，建立项目核心原则

目的

项目宪法是整个项目的"根本大法"，定义了核心原则、技术栈、质量标准和安全要求。 所有后续的规范和实现都必须遵循宪法中的原则。

核心内容

🎯
核心原则

代码质量标准、性能要求、设计哲学

🛠️
技术栈

编程语言、框架、数据库、工具链

🧪
测试标准

测试覆盖率、测试类型、验收标准

🔒
安全规范

数据隐私、认证授权、输入验证

示例内容

# 项目宪法：PDF AI 对话应用

## 核心原则
- 使用 TypeScript 严格模式
- 遵循 ESLint 和 Prettier 配置
- 保持函数单一职责

## 技术栈
- 前端: Next.js 14+ (App Router)
- 语言: TypeScript 5.x
- AI 集成: LangChain.js
- PDF 处理: pdf-parse 或 pdf.js

## 性能要求
- PDF 解析: 10MB 文件 < 5秒
- AI 响应: < 3秒

## 安全规范
- 仅允许 PDF 文件上传
- 最大文件大小 10MB
- 实施 CSP 防止 XSS
💡 最佳实践
  • 宪法应该在项目初期创建，并在整个项目生命周期中保持稳定
  • 每次添加新功能时，都要检查是否违反宪法原则
  • 宪法的修改需要团队讨论和共识

📝 步骤 2: 定义功能规范 (Specify)

/speckit.specify 添加用户认证功能，支持邮箱密码登录和 JWT token 每个新功能都需要创建规范

目的

将业务需求转化为结构化的功能规范，明确"做什么"和"为什么"，而不是"怎么做"。 规范面向业务利益相关者，使用非技术语言描述。

核心内容

📖
用户故事

按优先级排序的用户旅程，每个都可独立测试

✅
验收场景

Given-When-Then 格式的可测试场景

📋
功能需求

系统必须具备的具体能力

🎯
成功标准

可衡量的、技术无关的结果指标

规范质量检查

Specify 命令会自动生成质量检查清单，确保规范满足以下标准：

✓ 不包含实现细节（无语言、框架、API 名称）
✓ 聚焦用户价值和业务需求
✓ 需求可测试且无歧义
✓ 成功标准可衡量且技术无关
💡 独立可测试用户故事原则
  • 每个用户故事都应该可以独立开发、测试、部署
  • 按优先级排序（P1, P2, P3...），P1 是 MVP
  • 如果只实现 P1，应该仍然是一个可用的产品
  • 用户故事之间尽量减少依赖关系

📐 步骤 3: 技术规划 (Plan)

💡 宪法检查的重要性

宪法检查是一个"守门员"机制，确保所有技术决策都符合项目的核心原则。 如果设计方案违反了宪法，必须提供充分的理由，并记录在 Complexity Tracking 表中。

✅ 步骤 4: 任务分解 (Tasks)

/speckit.tasks 基于规范和计划生成可执行任务列表

目的

将设计文档转化为可执行的、独立的、可验证的任务列表。任务按用户故事分组， 支持并行开发和增量交付。

任务组织结构

Phase 1: Setup (共享基础设施)

项目初始化和基本结构

T001 项目结构 T002 依赖初始化 T003 [P] 代码规范配置
Phase 2: Foundational (阻塞前提条件)

核心基础设施，所有用户故事的前置依赖

T004 数据库schema T005 [P] 认证框架 T006 [P] API路由结构
⚠️ Checkpoint: 基础设施完成，用户故事可以并行开始
Phase 3+: User Stories (按优先级)

每个用户故事独立可测试、可交付

T010 [P] [US1] 测试 T011 [P] [US1] 模型 T012 [US1] 服务实现
🎯 Checkpoint: User Story 1 可独立验证 (MVP!)

任务格式说明

[ID] [P?] [Story] Description
  • [ID]: 任务编号（T001, T002...）
  • [P]: 可并行执行（不同文件，无依赖）
  • [Story]: 所属用户故事（US1, US2, US3...）
  • Description: 包含具体文件路径的任务描述

🚀 步骤 5: 执行实现 (Implement)

/speckit.implement 按任务列表执行所有任务，完成功能开发

目的

按照任务列表的顺序和依赖关系，逐步实现所有任务。实现过程遵循规范和计划文档， 确保代码质量和设计一致性。

实现流程

1
读取任务列表

解析 tasks.md，理解任务顺序和依赖关系

2
按顺序执行任务

尊重依赖关系，标记任务进行中/完成状态

3
代码质量保证

遵循宪法中的代码规范和安全要求

4
检查点验证

在每个检查点验证用户故事独立可用

5
增量交付

每个用户故事完成后可独立部署

实现原则

✓ 遵循规范

严格按照 spec.md 的用户故事和验收标准实现

✓ 遵循设计

按照 plan.md 的技术方案和数据模型实现

✓ 遵循宪法

遵守 constitution.md 的所有原则和标准

! 不过度设计

只实现规范中明确要求的功能，不添加额外功能

增量交付示例

Milestone 1: MVP

完成 User Story 1 (P1) - 最小可行产品

可部署 | 可演示 | 有价值
Milestone 2: Enhanced

完成 User Story 2 (P2) - 增强功能

包含 MVP | 更多价值
Milestone 3: Complete

完成所有用户故事 - 完整功能

全部就绪 | 生产部署
💡 实现最佳实践
  • 每个任务完成后提交代码，便于回滚和代码审查
  • 遇到阻塞及时沟通，不要盲目绕过
  • 保持代码整洁，避免"以后再重构"的想法
  • 定期在检查点验证，确保用户故事独立可用
