📚 PostgreSQL 基础概念

什么是 PostgreSQL？

  • 开源对象关系数据库系统 (ORDBMS) - 结合了关系型数据库和面向对象特性
  • ACID 完全兼容 - 保证数据一致性和可靠性
  • 支持多种数据类型 - JSON, XML, 数组, 几何类型等
  • 强大的扩展性 - 支持自定义函数、索引、数据类型

核心架构

🏗️ Client-Server 模型

客户端进程通过 PostgreSQL 协议与服务器通信

📦 多版本并发控制 (MVCC)

读写不阻塞，提供高并发性能

🔒 Row Level Security

行级安全策略，细粒度访问控制

⚡ Write-Ahead Logging

预写式日志，确保数据持久性和崩溃恢复

💡 Supabase 使用：本项目使用 Supabase，它是基于 PostgreSQL 的 BaaS 平台，提供自动化的 API 生成和认证系统。

🔤 PostgreSQL SQL 基础语法

1. 数据查询 (SELECT)

SELECT column1, column2, ... FROM table_name WHERE condition ORDER BY column_name ASC|DESC LIMIT count;

2. 数据插入 (INSERT)

INSERT INTO table_name (column1, column2, ...) VALUES (value1, value2, ...);

3. 数据更新 (UPDATE)

UPDATE table_name SET column1 = value1, column2 = value2, ... WHERE condition;

4. 数据删除 (DELETE)

DELETE FROM table_name WHERE condition;

5. 连接查询 (JOIN)

SELECT * FROM table1 INNER JOIN table2 ON table1.id = table2.table1_id;

⚡ PostgreSQL 高级特性

1. 行级安全策略 (RLS)

-- 启用 RLS ALTER TABLE user_pdfs ENABLE ROW LEVEL SECURITY; -- 创建策略：用户只能查看自己的数据 CREATE POLICY "Users can view own data" ON user_pdfs FOR SELECT USING (auth.uid() = user_id);

2. 触发器 (Triggers)

-- 创建自动更新 updated_at 的函数 CREATE OR REPLACE FUNCTION update_updated_at_column() RETURNS TRIGGER AS $$ BEGIN NEW.updated_at = NOW(); RETURN NEW; END; $$ LANGUAGE plpgsql; -- 创建触发器 CREATE TRIGGER update_timestamp BEFORE UPDATE ON user_profiles FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();

3. JSONB 数据类型

-- 存储和查询 JSON 数据 CREATE TABLE events ( id UUID DEFAULT gen_random_uuid() PRIMARY KEY, metadata JSONB ); -- 查询 JSON 字段 SELECT * FROM events WHERE metadata->>'type' = 'login';
🎯 项目应用：本项目使用 JSONB 存储用户元数据和安全日志详情，实现灵活的数据结构。

用户相关关系

每个认证用户对应一个用户资料

user_profiles → 1:N → user_pdfs

一个用户可以上传多个 PDF

对话相关关系

user_pdfs → 1:1 → pdf_conversations

每个配额定义有多条使用记录

user_profiles → N:N → quota_definitions

🔒 行级安全策略 (RLS)

项目中的 RLS 应用

1. 用户资料隔离

-- 用户只能查看自己的资料 CREATE POLICY "Users can view own profile" ON user_profiles FOR SELECT USING (auth.uid() = id); -- 管理员可以查看所有资料 CREATE POLICY "Admins can view all profiles" ON user_profiles FOR SELECT USING ( EXISTS ( SELECT 1 FROM user_profiles WHERE id = auth.uid() AND role = 'admin' ) );

2. PDF 文件隔离

-- 用户只能操作自己的 PDF CREATE POLICY "Users can delete own PDFs" ON user_pdfs FOR DELETE USING (auth.uid() = user_id);
⚠️ 安全注意：RLS 策略在数据库层面强制执行，即使应用层代码有漏洞，用户也无法访问不属于自己的数据。

3. 级联删除

-- 删除用户时自动删除相关数据 ALTER TABLE user_pdfs DROP CONSTRAINT user_pdfs_user_id_fkey, ADD CONSTRAINT user_pdfs_user_id_fkey FOREIGN KEY (user_id) REFERENCES auth.users(id) ON DELETE CASCADE;

💡 PostgreSQL 实用操作技巧

1. 索引优化

-- 创建索引加速查询 CREATE INDEX idx_user_pdfs_user_id ON user_pdfs(user_id); -- 复合索引 CREATE INDEX idx_quota_usage_user_quota_date ON quota_usage(user_id, quota_id, usage_date);

2. 常用查询示例

-- 获取用户今日 PDF 上传数量 SELECT COUNT(*) FROM user_pdfs WHERE user_id = $1 AND DATE(created_at) = CURRENT_DATE; -- 获取用户总存储使用量 SELECT COALESCE(SUM(file_size), 0) AS total_size FROM user_pdfs WHERE user_id = $1; -- 获取对话历史 SELECT role, content, created_at FROM conversation_messages WHERE conversation_id = $1 ORDER BY created_at ASC;

3. 事务处理

BEGIN; -- 插入消息 INSERT INTO conversation_messages (...) VALUES (...); -- 更新对话统计 UPDATE pdf_conversations SET message_count = message_count + 1 WHERE id = $1; COMMIT; -- 出错时使用 ROLLBACK;

安全建议

  • 始终使用参数化查询 - 防止 SQL 注入
  • 最小权限原则 - 应用使用受限用户，不使用超级用户
  • 启用 RLS - 数据库层面的安全防护
  • 定期备份 - 设置自动化备份策略
  • 监控日志 - 启用查询日志和慢查询日志

项目特定建议

💡 Supabase 特定：
  • 使用 anon 和 service role 两种密钥
  • RLS 策略是主要安全机制
  • 利用 Database Functions 实现复杂逻辑
  • 使用 Realtime 功能实现实时更新
⚠️ 注意：在生产环境中，避免直接在客户端执行复杂查询，应通过 API 端点封装。
